### src/main/java/com/ironhub/state/ProfileStore.java:58
**State file is written in place with no temp-file + atomic rename, and the loader silently discards the whole profile on a truncated file** (correctness)

- **Evidence:** save(): `Files.write(file.toPath(), json.getBytes(StandardCharsets.UTF_8));` writes directly to the final state.json. load() (lines 80–84): `catch (IOException | JsonParseException e) { log.warn("corrupt iron-hub state at {} — starting fresh", file, e); return new PersistedState(); }` — a truncated file parses as JsonParseException and yields a FRESH state, and the very next persist() overwrites the corrupt file with that fresh state, making the loss permanent. The truncation window is real in normal operation, not just power loss: IronHubPlugin.shutDown() line 160 `accountState.persistNow(); // flush` enqueues the write on RuneLite's shared client executor, and client exit (System.exit after plugin shutdown) does not await queued executor tasks — a close during the write truncates state.json.
- **Why it matters:** The in-code ponytail comment claims only 'a lost bank snapshot is recovered by opening the bank', but the same file carries goal seeds, the completion archive, saved setups/loadouts, farm runs and records, slayer/rumour history, and every pref map — none re-derivable from the client. One torn write = silent whole-account-history loss with a single log.warn.
- **Fix:** Three lines in save(): write to `state.json.tmp` in the same directory, then `Files.move(tmp, file.toPath(), StandardCopyOption.ATOMIC_MOVE, StandardCopyOption.REPLACE_EXISTING)` (fall back to a plain REPLACE_EXISTING move on AtomicMoveNotSupportedException). The reader then always sees a complete old-or-new file.
- **Verifier:** Line 58 is a direct in-place `Files.write(file.toPath(), json.getBytes(...))` and load()'s catch (`return new PersistedState()`) plus the next persist() makes a torn file permanent whole-profile loss; the ponytail comment's own rationale ('a lost bank snapshot is recovered by opening the bank') is factually wrong — persistNow (AccountState 2274–2345) snapshots goalSeeds, goalRecords, farmRunLog, slayerRecords, savedLoadouts etc., none re-derivable — and IronHubPlugin.shutDown:160 `accountState.persistNow()` does enqueue the write on the shared executor with nothing awaiting it before exit, so the temp+ATOMIC_MOVE fix is correct and conflicts with no project pattern (data-loss handling is explicitly exempt from the lazy-write shortcut).

- **Verifier:** Real: the loader callback (334-353) assigns data/optimizerService/panel with no staleness check, OptimizerService parks a `newSingleThreadExecutor` (OptimizerService:137) whose shutdown() only reaches the currently-referenced instance, and LoadoutLabModule.onThemeChanged does `lab.shutDown(); lab.startUp()` (251-253) — but the leak/resurrection requires a rare user action (theme flip or module toggle) landing inside the seconds-long dataset parse, so severity is low, not medium; the generation-counter fix in LoadoutLabPlugin.java is acceptable since that file already carries the three sanctioned decoupling edits.
