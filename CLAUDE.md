## The Iron Hub way ‚Äî patterns every new addition must follow

The design docs above define the product; these are the implementation patterns that emerged
building it. Every live module follows all of them ‚Äî new work must too, or it will read as
foreign next to the rest.

**Adding (or rebuilding) a module ‚Äî the recipe.**
1. Package `com.ironhub.modules.<name>`; a class implementing `IronHubModule` (`name()` = the
   nav-row label, `enabled()` = its `IronHubConfig` toggle, `startUp()`/`shutDown()` strictly
   symmetric: eventBus register/unregister, overlays + infoboxes add/remove, `tab.dispose()`).
   Register it in `IronHubPlugin.provideModules(...)`, add the config toggle
   (`modulesSection`), extend `ModuleLifecycleTest` (ctor call + module count) and confirm
   `PanelGalleryTest.navRowsExistForImplementedModules`.
2. Constructor-inject dependencies; every client-service param (`Client`, `ItemManager`,
   `OverlayManager`, `InfoBoxManager`, `Notifier`, `ClientThread`, `ConfigManager`) must
   tolerate null ‚Äî headless tests construct modules with nulls and the module must still
   start, build its tab and render. Event handlers run on the client thread; tab rebuilds
   happen ONLY on the EDT, and a tab's state/plan listener is ALWAYS
   `RebuildGate.install(this, this::rebuild)` (2026-07-17 freeze audit): hidden tabs mark
   dirty and rebuild once when next shown, visible tabs coalesce bursts ‚Äî a bare
   `invokeLater(this::rebuild)` listener rebuilds forever-invisible tabs on every state
   change, and with all hubs visited that was ~16 full rebuilds per kill. (Headless keeps
   always-rebuild semantics; build.gradle pins `java.awt.headless=true` for tests. A direct
   off-EDT rebuild races the listener and doubles every row; proven twice.)
3. Game state is read through `AccountState` (plus `watchVarbits`/`watchVarps`), never the
   Client directly from module logic ‚Äî that is what keeps modules unit-testable. Re-render on
   relevant change only (fingerprint-compare before rebuilding), never per-tick. Unbounded
   row lists are forbidden ‚Äî cap at ~50 with an honest "+ N more ‚Äî refine your search" line
   (the Bank grammar; clog/CA adopted it after rendering hundreds of sprite rows per rebuild
   measured as a freeze contributor).
4. Persistence: add fields to `PersistedState` + accessors on `AccountState` that
   `persist(); notifyListeners();`, and wire BOTH `activateProfile` (load) and `persist()`
   (snapshot). Gson keeps unknown/removed JSON keys harmless; field initializers are the
   defaults for old profiles. `persist()` is COALESCED on live clients (dirty flag, tick
   flush ~3s; logout/hop/profile-switch/shutdown call `persistNow()` ‚Äî it used to serialize
   the whole account on the game thread twice per kill) and synchronous headless; the
   ProfileStore executor owns both toJson and the write, fed a deep-copied snapshot.

**Static game content is a generated data pack, never Java constants.**
`tools/gen_<name>.py` ‚Üí `src/main/resources/data/<name>.json` + a draft-07 schema under
`data/schemas/` (DataPackTest fails the build for any pack without one). Generator
conventions: pin the upstream source (hub commit / RuneLite tag / wiki page), descriptive
User-Agent with repo+email, cache under `tools/.cache-*/` (gitignored), fail fast on any
unresolved constant or name, cross-validate references at generation time (e.g. quest tokens
against quests.json), and end with sanity asserts on counts. Requirement strings inside packs
use the graph's `parse()` form ‚Äî an unparseable string becomes a never-met manual gate, so a
pack test asserting every req parses non-manual is mandatory.

**The requirement graph is the only gating logic.** Never hand-roll met/unmet checks:
`Requirements.parse(...)`, `isMet(state)`, `describe()` for labels, `missing()` for unmet
leaves. Know the semantics before writing strings: `skill:` = hard/equip gate, `skillb:` =
boostable action gate, `item:` counts all ItemVariationMapping variants, `itemx:` = exact id
(tiers/imbues), `unlock:` keys must contain no colons, `any:a|b` with `&`-joined leaves =
alternative obtainment paths. DOMAIN-NOTES.md explains when each applies.

**"Track this" always means a synthetic Goal-planner goal** (standing user directive, applied
to gear, CA, diaries, collection log, clues, quests). Any track/target/+ affordance creates a
`<module>:<id>` goal through the UNIFIED seed system (Goals v2 G1, 2026-07-18): add a factory
method to `com.ironhub.state.GoalSeeds` (the family transform ‚Äî finished name/step
labels/proofs built ONCE at add time so the goal renders offline) and call
`state.addGoalSeed(GoalSeeds.<family>(...))` / `state.removeGoalSeed(goalId)` ‚Äî one persisted
`goalSeeds` map (`PersistedState.GoalSeed`: id/family/name/icon/steps/achieved/addedAt,
addedAt=0 means migrated-date-unknown), one `toGoal(seed)` in GoalPlannerModule, one loop in
`allGoals`, one generic branch in `removeGoal`; proof-marking modules scan
`state.goalSeedIds(family)`. The five legacy per-family maps (caGoals/diaryGoals/clueGoals/
clogGoals/customGoals) migrate in `activateProfile` via the same factories (write-through
once; GoalSeedMigrationTest pins byte-identical output). Achieved proof = a `<prefix>_<id>`
unlock flag the module marks from live data (and immediately at add time when already
satisfied). Requirements in the seed become honest planner steps. Never build a module-local
tracked list ‚Äî dashboard picks planner goals up for free. **Goals v2 G2 added four more
providers (all four "+ Goal" affordances on existing tabs, no new modules):** PoH
(`poh:<tierId>` ‚Äî proof `unlock:pohtier_<sanitized>`, tier ids carry a colon so the key MUST
`replace(':','_')` or the graph's colon-split unlock: parse truncates it ‚Äî `GoalSeeds.pohProofKey`;
marked by PohModule's `markPohGoalProofs` listener), QoL (`qol:<id>` ‚Äî achieved by OWNING the
unlock via `item:`/`any:` ownership proof, NOT "reqs met"; the prose reqs are display steps),
Bank (no new family ‚Äî the skill-target `+ Goal` builds the SAME `custom:skill:<skill>:<lvl>`
id the planner search builds so they dedupe; capped 2..99 since `skill:` gates on real level),
Supplies (`supply:<itemId>` one-shot "stock N √ó item", proof `item:<id>:<N>:<name>`, re-addable).
**Completion archive (G2):** `PersistedState.GoalRecord` (goalId/name/family/addedAt/completedAt/
estimatedHours/hoursAtCompletion) in a capped-200 `goalRecords`, upserted by goalId; the
detector lives in `GoalPlannerModule.detectCompletions` (planner thread, in replanNow): a
SELECTED goal flipping unachieved‚Üíachieved records once, dated now() only if seen unachieved
this session (`seenActiveGoalIds`), else `completedAt=0`="detected"; the first pass never
re-records an already-archived goal (login replay keeps its date); removal never records.
`estimatedHours` captured by `GoalPlannerModule.captureEstimate` at add time (0=unknown, module `+` clicks leave 0; previewMerge deleted 2026-07-20).

**Combining or porting other plugins ‚Äî the playbook** (used for Log Adviser, Time Tracking,
Time Tracking Reminder, Easy Farming; Loadout Lab was imported whole only on explicit user
direction). Default is PORT, not import: fetch the source at the plugin-hub pinned commit
(`raw.githubusercontent.com/runelite/plugin-hub/master/plugins/<name>` ‚Üí repo+commit) and
read it fully before writing anything. Keep their proven math, data tables and detection
tricks byte-faithful; replace their state/config/UI layers with AccountState, the requirement
graph and UiTokens. Convert their data to a generated pack. Vendored upstream CODE (when
package-private visibility forces it) is emitted by a generator with writes stripped, and
pinned by a reflective parity test against the classpath client so client updates fail CI
with a regenerate command (`TimetrackingParityTest`, `tools/gen_timetracking.py`). Reading
another plugin's config group read-only is hub-safe precedent; never write another plugin's
group. Always: `licenses/<name>-LICENSE`, class-header attribution, README credit (BSD-2
requires it).

**The polish bar ‚Äî what "done" means here.**
- Full `./gradlew build` green before every commit; never commit or push red.
- Every new UI surface writes a render PNG from a test (`build/reports/*.png`) ‚Äî and you must
  actually LOOK at the render before calling the work done; that habit has caught doubled
  rows, truncation, and spacing bugs the assertions missed.
- Honesty is a feature: unknown data renders as "?"/NaN/silence, never an invented name, rate
  or time; distinguish banked vs carried vs raw quantities explicitly; copy states its
  provenance ("as of last log open", "predicted"). Negative-path tests are expected
  (never notify on login replay, never flash Done on goal removal, unknown stays unknown).
- Tests use the shared fixtures (`StateFixture`, `TimetrackingFixture`,
  `InMemoryConfigManager` pattern: real logic over in-memory doubles, seeded byte-for-byte in
  the same format the live source writes) ‚Äî not per-test mock forests. Derive game constants
  in tests by sweeping the real decoder/pack, never hardcoded magic values.
- UI: styles only from `UiTokens`; user-visible strings pass GlyphSafetyTest (ASCII plus
  `¬∑ ‚Ä¶ ‚Äî √ó`); 14 px text floor; semantic status colors never repurposed; detail that doesn't
  fit 225 px goes in tooltips; Swing specifics ‚Äî `mousePressed` not `mouseClicked`, no nested
  scroll panes, explicit `LEFT_ALIGNMENT`, `getMaximumSize()` override on BoxLayout tabs, and
  UI-less custom JComponents MUST override `getMinimumSize()` (default = current size = 0√ó0
  pre-layout, and BoxLayout derives row alignment from child MINIMUMS ‚Äî an all-zero-min row
  cuts children to half height top-anchored in the client, while SwingRender's multi-pass
  layout self-heals it; pin such bugs with a single-pass client-style mount test).
- Commits: one coherent slice per commit with an explanatory body, ending
  `Co-Authored-By: Claude Fable 5 <noreply@anthropic.com>`; docs (CLAUDE.md state, README,
  DOMAIN-NOTES for any new game-data knowledge) updated in the same arc.

**Standing user gates (do not relitigate):** Plugin Hub submission happens only on the
user's explicit word ‚Äî never prepare it unprompted (when they say go: repo needs a LICENSE
file and a hub-sandbox build check). External HTTP is opt-in or user-initiated only
(build-time generators are fine). Meta-data-pack regenerations (methods/rates refreshes,
the quarterly-ish cadence agreed for Goals v2) are NEVER implemented without first laying
the proposed changes out to Luke (2026-07-18). Bug reports get root-caused ‚Äî verify pack data and the
upstream source before touching code; the reported symptom is often a data error. Match
reference plugins from their actual source, not from memory.
